# Dependencies And Convergence

Every prerequisite is stored once by its blocked consumer. Dependencies never transfer ownership. Reverse `blocks` and impact views are derived navigation with a source locator and observation time; do not maintain a second mutable direction.

## Ownership Partition

- A Work Item owns its direct implementation prerequisites.
- A Leg records only external facts, authorities, evidence, or other Leg outcomes not already represented by a Work Item edge. Cross-Leg ordering may be derived from Work Item edges.
- A Map owns cross-Feature prerequisites and cross-Map prerequisites needed by its blocked Map or Feature. It does not copy every internal Work Item edge.
- A cross-Map edge cites the provider Atlas/Map/Feature and exact endpoint, the provider Decision observed, satisfaction test and authoritative source, freshness/compatibility assumptions, invalidators, unavailable impact, convergence use, and revalidation.

Provider acceptance, implementation, completion, or a broad passing suite does not itself satisfy a consumer-specific endpoint. The named satisfaction test must establish the required Contract/output/evidence/system condition.

## Semantic DAG Inspection

Edges name concrete delivery prerequisites: an interface before its consumer integrates, a data-preserving migration before authority moves, compatibility behavior before an old path retires, material evidence before broad dependence, or an integrated outcome before another consumer relies on it. Ordering preference, convenience, and team habit are not edges.

Inspect each local Work Item graph, Feature Leg graph, and Map Feature graph over the accepted/candidate nodes visible to the decision context plus each concrete declared endpoint. Provider internals beyond an endpoint remain opaque unless separately visible. Return:

- `acyclic` when this bounded view is inspectably acyclic;
- `dependency_cycle` with the smallest visible cycle;
- `dependency_acyclicity_unknown` when a node or endpoint is inaccessible, stale, ambiguous, or unresolvable.

Cycle or unknown prevents exact candidate acceptance. Clarify the seam, merge an indivisible responsibility, add one owned convergence, or return an insufficient Contract to Blueprint. Never claim global cross-Atlas acyclicity, infer hidden provider edges, or require a graph engine, locks, leases, reservations, or universal revision protocol.

Parallel-frontier and critical-gate descriptions are optional human explanations when useful, not computed outputs.

## Behavioral Feedback Is Not A Delivery Cycle

Example: planner/material → issuance → local runtime → reconciliation → release proof is an acyclic delivery graph. Reconciliation emits a canonical fact consumed by a *future planning run*. Record that relationship as `behavioral feedback — not blocked by`, naming producer fact, future consumer behavior, and temporal boundary. Do not add an edge from reconciliation back to the already-built planner.

## Convergence

Every convergence names exactly one Feature or Map owner, input endpoints and source owners, compatibility assumptions, integration/observation boundary, joined evidence, intended independent checker, E2E/security/cleanup allocation, failure route, and downstream consumer.

Feature convergence integrates its Legs. Map convergence owns cross-Feature or cross-Map joint proof and references rather than copies source evidence. When joint proof needs implementation, place that work in an identified Feature Leg while the Map retains proof ownership.
